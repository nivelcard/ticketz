# Regras de documentação oficial — Ticketz

**Status:** permanente, aplicável a todas as implementações  
**Fonte de verdade:** `docs/MANUAL_PLATAFORMA.md`  
**Rule Cursor:** `.cursor/rules/documentation-rules.mdc` (`alwaysApply: true`)

---

## Objetivo

Transformar o `MANUAL_PLATAFORMA.md` na documentação técnica oficial da plataforma.

A partir de agora, **toda alteração estrutural** realizada no sistema deverá **obrigatoriamente atualizar a documentação antes da conclusão da tarefa**.

Esta rule é aplicada automaticamente pelo agente em todas as implementações futuras.

---

## Escopo de alterações cobertas

Sempre que houver qualquer alteração relacionada a:

- arquitetura
- banco de dados
- migrations
- modelos
- serviços
- APIs
- endpoints
- IA
- agentes
- RAG
- embeddings
- prompts
- filas
- integrações
- frontend administrativo
- infraestrutura
- autenticação
- permissões
- deploy
- variáveis de ambiente
- workflows
- pipelines
- dependências externas
- organização de pastas

---

## Procedimento obrigatório

O agente deverá:

1. **Localizar** a seção correspondente dentro de `docs/MANUAL_PLATAFORMA.md`
2. **Atualizar** a documentação no manual
3. **Criar nova seção** se a funcionalidade for nova
4. **Atualizar diagramas Mermaid** quando necessário
5. **Atualizar fluxos** afetados
6. **Atualizar tabelas** (models, env vars, endpoints, filas)
7. **Atualizar exemplos** (requests, configs, comandos)
8. **Atualizar dependências** entre módulos
9. **Atualizar o roadmap** (§22 + `docs/roadmap.md`)
10. **Atualizar limitações conhecidas** (blocos Auditoria + §41)
11. **Atualizar dívidas técnicas** (§41)
12. **Atualizar riscos arquitetônicos** (§43)
13. **Atualizar o histórico** (`docs/changelog.md` + versão no cabeçalho do manual)

Atualizar também o **índice temático** correspondente em `docs/<tema>.md`.

---

## Validação obrigatória

Antes de concluir qualquer implementação, verificar:

- [ ] O código foi atualizado
- [ ] O manual foi atualizado
- [ ] Os diagramas representam o código atual
- [ ] Os exemplos continuam válidos
- [ ] As variáveis de ambiente estão corretas
- [ ] Os endpoints documentados existem
- [ ] Os serviços documentados existem
- [ ] As tabelas documentadas existem
- [ ] As migrations documentadas existem
- [ ] O roadmap continua consistente

---

## Relatório final

Todo relatório técnico deverá informar:

- Se houve atualização do manual
- Quais seções foram alteradas
- Quais diagramas foram modificados
- Quais novos componentes foram documentados
- Se a documentação permanece sincronizada com o código
- Se `docs/changelog.md` foi atualizado

Template:

```markdown
## Documentação

| Item | Detalhe |
|------|---------|
| Manual atualizado | sim / não |
| Seções alteradas | §X, §Y, … |
| Diagramas modificados | §25, §26, … / nenhum |
| Novos componentes documentados | … / nenhum |
| Sincronizado com código | sim / não |
| Changelog | docs/changelog.md — sim / não |
```

## Estrutura da documentação

```
docs/
├── MANUAL_PLATAFORMA.md      ← visão geral e referência completa (fonte de verdade)
├── .documentation-rules.md   ← este arquivo
├── architecture.md           ← índice: arquitetura
├── ai.md                     ← índice: IA
├── rag.md                    ← índice: RAG
├── database.md               ← índice: banco
├── api.md                    ← índice: endpoints
├── deployment.md             ← índice: deploy
├── frontend.md               ← índice: frontend
├── backend.md                ← índice: backend
├── integrations.md           ← índice: integrações
├── roadmap.md                ← índice: evolução
└── changelog.md              ← histórico de alterações da documentação
```

Os arquivos temáticos (`architecture.md`, `ai.md`, etc.) são **índices** que apontam para seções do manual. Conteúdo extenso fica centralizado no `MANUAL_PLATAFORMA.md` para evitar divergência.

## Versionamento do manual

Formato do cabeçalho: `Versão: X.Y`

- **X** — mudança estrutural maior (nova seção, refactor arquitetural)
- **Y** — correção ou complemento dentro da mesma estrutura

Toda alteração estrutural exige entrada em `docs/changelog.md`.
